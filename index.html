Third Down Analysis: CMPS 3160¶

Hi, my name is Sam Bruchhaus. As someone who is currently pursuing a career in Sports Analytics, I am excited to finally get the opportunity to flesh out a full-length project. Though I have worked on several smaller projects, namely an NFL Draft Quarterback Performance Predictor and an NBA Salary Predictor, this will be my first project using generally accepted data science techniques and tools I have learned in CMPS 3160 at Tulane University.

The goal of this project is to explore NFL Play-By-Play Data. More specifically, the possible research questions for this project can be broken down into three major question types, all of which deal with play-calling decisions on third down:

  • “3rd and Medium”: On 3rd Downs with a distance of 4 yards, 5 yards, or 6 yards (also known as “3rd and Medium”), are run plays or pass plays more efficient? Does this vary by team? If so, is there some pattern that governs which teams are better at passing versus running?
  • “Playing for 4th and Short”: On 3rd Downs with a distance of greater than 3 yards, do teams tend to go for the 1st down or play for a manageable 4th down? What factors determine this strategy? Is this a viable strategy in any case?
  • “3rd Down in the Hot Zone”: For the purposes of this project, the “Hot Zone” will be defined as plays run from the 3-Yard Line to the 10-Yard Line going in to score a touchdown. On 3rd Downs in this area, do teams tend to go for the touchdown or get closer to the goal line? How does this decision-making change at the end of close games? Are there any patterns on how this decision has been managed in past years?

Using a .csv files with Play-by-Play Data from 2018 (link below) and 2019 created using a package called nflscrapR and posted on Github, I will be able to provide insight into some of these questions. More information on the data including the current capabilities and future possibilities can be found below.

I plan to work on this project alone. The reasoning for this is two-fold. First, I want to use this as an opportunity to build my analytical knowledge of the football. However, I was worried that if I partnered with others, my role would primarily be to lend credence or caution to the findings of this project based on my past experiences. In order to ensure that I am primarily a data scientist and secondarily a football player, I am working alone. Second, I would like to use this project as a sample for job applications in sports. Because I know sports organizations are very thorough in their research, I wanted to make sure that the organizations know that I am capable of all the work contained in this project, and that I did not specialize in one part or another.

I plan to complete this project using jupyter notebooks with a possible addition of some better visualization services. All updates will be posted to sbruchhaus.github.io. Any questions can be directed to sbruchhaus@tulane.edu.

Tidying Data for Use in Further Milestones¶

Now that the Data is relatively cleaned of information not necessary to our end goal, we can tidy the data. The following checks to ensure tidy data are as follows:

  • Column headers are variable names, not values.

  • Multiple variables are not stored in one column.

  • Variables are not stored in both rows and columns.

  • Multiple types of observational units are not stored in the same table.

  • A single observational unit type is not stored in multiple tables.

Column headers are variable names, not values.¶

There does not appear to be any column headers that are values. All describe the play in some way.

Multiple variables are not stored in one column.¶

There appear to be one column in which multiple variables are stored in a column. Below is the code to remove the following:

yrdln: Contains both Qualitative Data (which team's side of the field) and Quantitative Data (The Yardline)

In [5]:
third_downs_clean = third_downs_clean.drop(columns="yrdln")

Variables are not stored in both rows and columns¶

It does not appear that any variables are stored in the rows. All values in the rows are relevant to their variables, which describe the observation, i.e. the play.

Multiple types of observational units are not stored in the same table¶

The initial data does a poor job of seperating tables by observational unit. There are three major types of data in this table that we plan to use.

Qualitative Descriptors -- can apply to play_ids, game_ids, and play_types

Quantative Descriptors in a set -- really only apply to play_ids

Quantitative Values not in a set -- best describe play_ids and play_types

Checking Overarching Tidy Data Principles to Ensure Tidiness¶

  1. Each variable forms a column and contains values -- Yes, all variables are overarching measurements/attributes and are stored in columns. Each has values which are specific measurements or attributes.
  1. Each observation forms a row -- Yes, in each table other than "qual_game", the individual play is the observation that forms the row. "qual_game" is an extension of qual_play. It contains data regarding each indvidual game, and the game is the observation.
  1. Each type of observational unit forms a table -- Yes. The observational units are as follows:
    • qual_play, qual_game, qual_passes, qual_runs = Qualitative, Categorical Objects
    • quant_des_play = Quantitative, Discrete Data
    • values_play, values_passes = Quantitative, Continuous Data
